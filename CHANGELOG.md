# Changelog

All notable changes to the iAqualink Robots Home Assistant integration are documented here.

## Unreleased

### Changed

- **i2d-family cloud behaviour extracted into `protocols/i2d.py` — completes the R25 epic (story R25-i2d, refactor — folds R27 god-function).** Fifth and final per-device-type protocol extraction; with R25-i2d landing, every supported device family has a dedicated protocol class and `_DEVICE_PROTOCOLS` is no longer a partial dispatch table. i2d is the outlier of the 5 families — it talks to the iAqualink cloud over **HTTP POST** instead of the WebSocket shadow stream every other family uses. Commands are encoded as hex `params` strings in HTTP bodies (`request=0A1240` to start, `request=0A1210` to stop, etc.); status responses are raw 18-byte hex frames the parser unpacks byte-by-byte into a 14-field `ParsedI2DFrame`. R25-i2d also folds the R27 "god-function" cleanup the team review flagged: pre-R25 the parser was a 138-line method body in `coordinator.py` with three inline lookup tables (`state_map`, `error_map`, `mode_map`) rebuilt on every call and side-effects (HTTP fetch, `self._fan_speed`, `self._activity`) mixed in with the byte-unpacking logic. R25-i2d splits that into a **pure** `parse_i2d_payload(hex_str: str) -> ParsedI2DFrame` function (no I/O, no logging, no globals beyond the now-module-level `STATE_MAP` / `ERROR_MAP` / `MODE_MAP` constants — fully unit-testable by passing hex strings) plus `I2DProtocol.parse_status` which wraps the pure parser and lifts the parsed dataclass into the coordinator's `result` dict + applies the per-family side-effects. The i2d-unique "any non-zero `error_code` flips `result["activity"]` to `"error"`" mapping (none of the 4 WebSocket families do this — they keep activity at idle/cleaning/returning and route errors through `error_state`) is preserved and explicitly tested. The `walls_only` (plural, i2d waterline mode) vs `wall_only` (singular, VR/cyclobat dedicated wall-scrub) distinction from CLAUDE.md is locked at the protocol layer too — `I2DProtocol.fan_speed_codes()` contains `walls_only` and explicitly NOT `wall_only`, with a dedicated test that documents the parity contract. The shape difference between i2d (HTTP body dict) and the 4 WS families (WebSocket envelope dict) is intentional and the abstract base's `dict` return type accommodates both; the coordinator's command sites dispatch by `device_type` — i2d branches send via `post_command_i2d(url, request)` (URL helper `I2DProtocol.i2d_url(client)` is i2d-only, not on the abstract base), other branches via `set_cleaner_state(request)`. Cyclonext-style "no pause command" applies here: `pause_payload` returns the same envelope as `stop_payload` because pre-R25 `pause_cleaning` for i2d delegated to `stop_cleaning`. `extract_fan_speed_from_response` returns `None` for i2d since HTTP responses don't carry a confirmed-mode echo the way WS replies do; the polled status frame surfaces the cloud's actual mode via `parse_status`. **`_update_i2d_robot` is reduced from 138 lines to 7 statements** (HTTP fetch + delegate to `i2d.parse_status`); the spec AC #2 budget was ≤ 30, the actual reduction is ~95%. The 4 i2d command sites in `coordinator.py` (`start_cleaning` / `stop_cleaning` / `return_to_base` / `_set_i2d_fan_speed`) all route through `get_protocol("i2d_robot")` now. **Bonus cleanup**: `_extract_fan_speed_from_response`, which used to hand-code the per-family decode in 4 if/elif branches, collapses to a single `protocol.extract_fan_speed_from_response(response_data, requested_fan_speed)` call since every family now has a protocol. New `tests/protocols/test_i2d.py` (39 tests + parametrized variants) covers byte-identical envelopes including the HTTP-body shape distinct from the WS families, the URL helper, the `walls_only` parity guard, the pure parser across 7 state codes + 10 error codes + 9 mode codes + canister-bit + time/uptime/hours/IDs + 2 fallback-on-unknown cases + 2 hex-malformation cases, the module-level constants contract (spec AC #1), 6 parse_status behavioural cases including the i2d-unique error→activity mapping, the just-started-cleaning state-code 0x02 → cleaning mapping, the non-status-request bail-out, and the malformed-hex update_failed path, plus AST dispatch guards for the 4 coordinator command sites + a `_update_i2d_robot` body-length budget (≤ 30 statements per spec AC #2). `mypy.ini` extended for `protocols.i2d`. Dispatch: `_DEVICE_PROTOCOLS["i2d_robot"] = I2DProtocol()` — **all 5 R25 families now have protocols**; future regression test guards in `tests/protocols/test_<family>.py` confirm `get_protocol(family)` returns a protocol (not `None`) for every supported family. No user-facing behaviour change — cloud sees the same HTTP body / WS envelopes; the parser produces the same `result` dict.

- **Cyclonext-family cloud behaviour extracted into `protocols/cyclonext.py` (story R25-cyclonext, refactor).** Fourth of five per-device-type protocol extractions. Cyclonext is the most minimal of the WebSocket families: only 2 fan speeds (`floor_only` / `floor_and_walls`), only 2 activity states (`cleaning` / `idle` — no `returning` since the family has no return-to-base), and the only one that nests its cloud state under `equipment.robot.1` (the `.1` suffix is distinct from every other family's `robot` key) and uses `mode` instead of `state` for activity decoding. `CycloNextProtocol.return_to_base_payload` and `clear_desired_payload` both inherit the abstract base's `None` default — locking the "no RTB for cyclonext" invariant the pre-R25 inline branch list `["vr", "vortrax", "cyclobat"]` enforced by exclusion. The set_fan_speed payload nests under `robot.1` with the `cycle` field carrying a STRING value (the cloud rejects non-string variants on some firmware revisions per the comment that pre-issue-#76 reported as the silent-no-op trigger). The fan-speed encoding map is locked to exactly 2 keys (`floor_only` / `floor_and_walls`) — a regression test explicitly asserts that the legacy display-name keys (`"Floor only"` / `"Floor and walls"`) and unsupported translation keys (`wall_only` / `walls_only` / `smart_floor_and_walls`) do NOT appear in `fan_speed_codes()`. Issue #76 originally reported "fan speed no longer propagates from HA to AquaLink" on RE 4400 iQ because the pre-fix dispatch map used display-name keys and the `.get()` lookup always returned None; the protocol layer locks the snake_case translation-key contract so a future refactor can't silently regress. The parser is the simplest of the WebSocket families — reads from `equipment.robot.1` (rejects fallback to `robot`), 2-state activity decoder, scaled canister, error_state from nested `errors.code` (not flat `errorState`), `total_hours` from `totRunTime` with silent zero fallback for older models that don't expose the field. 5 coordinator command sites (start/stop/pause + `_set_other_fan_speed` + `_extract_fan_speed_from_response`) and the `_update_cyclonext_robot_data` parser delegate now route through `get_protocol("cyclonext")`. New `tests/protocols/test_cyclonext.py` (29 tests + parametrized variants) covers byte-identical envelopes including the `robot.1` key and `mode`-int / `cycle`-string distinctions, the only-2-fan-speeds contract (with the issue #76 regression guard checking forbidden keys), the only-2-activity-states invariant (explicit assertion that `result["activity"] != "returning"` for any non-1 mode value), parser handling of missing-`robot.1` payloads, the `errors.code` nested-dict path, the `totRunTime`-with-zero-fallback contract, and the standard AST dispatch + body-budget guards. `mypy.ini` extended for `protocols.cyclonext`. Dispatch: `_DEVICE_PROTOCOLS["cyclonext"] = CycloNextProtocol()`; only i2d_robot remains inline-encoded (R25-i2d pending — and that one will be different because i2d uses HTTP POST not WebSocket envelopes). No user-facing behaviour change.

- **Cyclobat-family cloud behaviour extracted into `protocols/cyclobat.py` (story R25-cyclobat, refactor).** Third of five per-device-type protocol extractions. Cyclobat is the first family in the R25 series where the abstract base's flexibility matters — it diverges from VR/Vortrax in three meaningful ways: (1) lifecycle commands use `setCleaningMode` action with `{"main": {"ctrl": N}}` nesting and explicit `namespace="cyclobat"` (not VR/Vortrax's flat `setCleanerState` + `robot.state`); (2) the RTB anomaly — cyclobat's `return_to_base_payload` is the one command that does NOT use the `main.ctrl` form, it uses the same flat `setCleanerState` + `robot.state = 3` shape as VR/Vortrax (the pre-R25 inline comment said "the cloud accepts it today and changing it would be a behavior change"; the protocol preserves this exactly and the test locks it so a future "fix the inconsistency" PR can't silently change the cloud contract); (3) the fan-speed encoding uses `main.mode` with STRING values (cloud expects strings, unlike VR's int `prCyc`) and a different number assignment — VR's `0=wall_only` becomes cyclobat's `3=wall_only`, with `0` and `1` swapped between floor_only and floor_and_walls. The parser is the most divergent of the protocol family — cyclobat exposes a much richer state surface (battery sub-block with level / charge state / cycles / warning code, stats sub-block with totRunTime / diagnostic / temperature / lastError, lastCycle sub-block with cycleNb / duration / mode / endCycleType, cycles sub-block with per-cycle-type durations and a `firstSmartDone` flag) reading from entirely different cloud paths than VR/Vortrax (`main.state` not `robot.state`; `stats.tmp` not `sensors.sns_1.val`; `stats.totRunTime` not `robot.totalHours`). For that reason `CycloBatProtocol.parse_status` does NOT use the R29-fold `apply_common_robot_fields` helper — its field paths don't overlap with VR/Vortrax. The cyclobat cycle_duration picker uses a 4-branch switch on `lastCycle.endCycleType` (0=floorTim, 1=floorWallsTim, 2=smartTim, 3=waterlineTim) — also locked in tests. 6 coordinator command sites + the `_update_cyclobat_robot_data` parser delegate now route through `get_protocol("cyclobat")`; the legacy method is a 2-statement stable seam matching the VR/Vortrax shape. New `tests/protocols/test_cyclobat.py` (29 tests + parametrized variants) covers byte-identical envelopes including the RTB anomaly, the string-not-int fan-speed encoding contract, the 4-branch cycle_duration picker (one test case per endCycleType value), battery field surfacing, version/serial extraction, the temperature-from-stats-tmp invariant (proves cyclobat doesn't reach for VR's sensors.sns_1.val path), and an AST dispatch guard for all 6 cyclobat command sites + the parser delegate. `mypy.ini` extended for `protocols.cyclobat`. Dispatch table: `_DEVICE_PROTOCOLS["cyclobat"] = CycloBatProtocol()`; cyclonext and i2d_robot still return `None` from `get_protocol()` until their R25 stories land. No user-facing behaviour change.

- **Vortrax-family cloud behaviour extracted into `protocols/vortrax.py` + shared parse helper (story R25-vortrax, refactor — R29 folded in per the team review).** Second of five per-device-type protocol extractions, following the R25-vr template. Pre-R25-vortrax the Vortrax family was inline-encoded across the same six if/elif chains as VR — and the two parsers were copy-paste twins of each other for ~40 lines (the temp/activity/canister/errorState/totalHours block). R25-vortrax does three things in one commit: (1) introduces `protocols/common.py` with `apply_common_robot_fields(client, robot_data, result)` — lifted verbatim from the duplicated section of the two parsers, owns the shared field mapping in a single source of truth; (2) refactors `VRProtocol.parse_status` to delegate that block to the shared helper (drops ~30 lines from `vr.py`); and (3) adds `VortraxProtocol` mirroring the VR shape — same `setCleanerState` + `robot.state` lifecycle envelopes (1=start, 0=stop, 2=pause, 3=return-to-base), same `setCleaningMode` + `prCyc` fan-speed encoding (the cloud accepts all 4 modes; `vacuum.py`'s 2-mode `_fan_speed_list` is the UI restriction), with Vortrax-specific bits in `parse_status` (the `product_number` from `eboxData.completeCleanerPn`, the stepper-less time-math path on `_calculate_times`). `VortraxProtocol.clear_desired_payload` deliberately inherits the abstract base's `None` default — PR #94's auto-restart bug is VR-only today; if a Vortrax operator reports the symptom, override there. The 5 coordinator command sites with Vortrax branches (start/stop/pause/return-to-base/`_set_other_fan_speed`) now route through `get_protocol("vortrax")` — matching the R25-vr shape — and `_extract_fan_speed_from_response`'s Vortrax branch collapses from 8 lines of inline decode to a 3-line protocol delegate. The legacy `AqualinkClient._update_vortrax_robot_data` is now a 2-statement stable seam delegating to `VortraxProtocol.parse_status`, same shape as `_update_vr_robot_data`; both will collapse once R25-cyclobat / R25-cyclonext / R25-i2d land and R26 (module split) does the final dispatch cleanup. Dispatch: `_DEVICE_PROTOCOLS["vortrax"] = VortraxProtocol()`; the test asserts cyclobat/cyclonext/i2d_robot still return `None` so a future PR populating the table without migrating the coordinator branches fails CI immediately. New `tests/protocols/test_vortrax.py` (27 tests + parametrized variants — mirrors test_vr.py) covers byte-identical envelopes, the prCyc bidirectional decode, parser handling of malformed payloads / 4 activity states via shared helper / canister scaling / `product_number` extraction + None fallback / never-cleaned `cycleStartTime=0` / the Vortrax-specific invariant that the parser does NOT touch `stepper` / `stepper_adj_time` / `fan_speed` keys (those are VR-only), AST dispatch guards for the 5 Vortrax command sites, and a body-length budget for `_update_vortrax_robot_data` (≤ 3 top-level statements). New `tests/protocols/test_common.py` (15 tests) covers the shared helper in isolation — temperature `.val` → `.state` → `'0'` fallback ladder, activity mapping for all 5 cloud-state values (1/3/0/2/-1), canister 0-1 → 0-100 scaling for 4 corner cases, missing-field defaults for canister/error_state/total_hours, and the structural invariant that the helper writes EXACTLY 5 keys and doesn't touch anything else. The mypy.ini bootstrap-ignore set is extended for `protocols.common` and `protocols.vortrax` (P6 will ratchet them off individually). No user-facing behaviour change — cloud sees dict-equal envelopes, parsers produce the same `result` dict, the deferred-fields invariant means VR-specific keys still only appear on VR parses and vice versa.

- **VR-family cloud behaviour extracted into `protocols/vr.py` strategy class (story R25-vr, refactor).** First of five per-device-type protocol extractions (the R25 epic). Pre-R25 the VR family's cloud encoding lived inline across six if/elif chains in `coordinator.py` — the start/stop/pause/return-to-base/clear-desired-state envelopes (5 sites), the `_set_other_fan_speed` `prCyc` map, the `_extract_fan_speed_from_response` decoder, and a 120-line `_update_vr_robot_data` parser. Adding a new VR firmware variant or fixing a parse bug meant touching seven separate sites and risking drift between them. R25-vr introduces a new `custom_components/iaqualink_robots/protocols/` package with `base.py` (the `RobotProtocol` abstract base — namespace + 6 envelope builders + 2 cloud-encoding helpers + `parse_status` parser), `vr.py` (`VRProtocol` concrete implementation owning the VR family's `prCyc` map and the lifted-verbatim parser body), and `__init__.py` (the `_DEVICE_PROTOCOLS` dispatch table — `get_protocol(device_type)` returns the matching instance, or `None` for the four families still inline-encoded until their R25 story lands). The 6 coordinator command sites now read `if self._device_type == "vr": request = get_protocol("vr").start_payload(self)` (similar for the other 5 methods) — the inline `_build_state_request(...)` form is gone from the VR branches. The legacy `AqualinkClient._update_vr_robot_data` method survives as a 2-statement stable seam (delegates to `get_protocol("vr").parse_status(self, data, result)`) because both the polled path (`fetch_status`) and the push path (`_apply_realtime_envelope`) call it by name — that dispatch will collapse once all five R25 stories land (story R26 — module split). Net `coordinator.py` reduction: **134 lines removed, 51 added** (-83 net), with the parser body relocated to `vr.py` so the per-family ownership is explicit. `test_vr.py` (35 tests + parametrized variants under `tests/protocols/`) locks the AC #4 contract: every envelope builder produces a dict-equal envelope to the pre-R25 inline form (one parametrized test case per envelope shape × 4 fan-speed keys = 14 envelope cases), the `prCyc` ↔ fan-speed-key bidirectional map covers all 4 VR speeds plus the unknown-cloud-value fallback (5 cases for each direction), the parser handles missing-payload / malformed-payload / 4 activity states / 5 fan_speed states / canister scaling / temperature `.val` vs `.state` fallback / never-cleaned (cycleStartTime=0) edge case / missing-stepper defaults, an AST-level dispatch guard fails CI if any of the 6 coordinator command methods bypasses the protocol for the VR branch and re-inlines `_build_state_request(...)`, and a body-length budget (`_update_vr_robot_data` ≤ 3 top-level statements) catches a future regression that re-inlines the 120-line parser. The four other families (vortrax / cyclobat / cyclonext / i2d) still use the inline if/elif branches — each gets its own R25-`<family>` story; the dispatch table returns `None` for them and the matching test asserts that (so a future PR that populates the table without migrating the coordinator branches is caught immediately). The `wall_only` vs `walls_only` distinction documented in CLAUDE.md is locked at this layer too — `VRProtocol.fan_speed_codes()` contains `wall_only` (singular, VR dedicated wall-scrub) and explicitly NOT `walls_only` (i2d waterline-only). No user-facing behaviour change; the cloud sees the same envelopes (verified dict-equal) and parsers produce the same `result` dict.

- **Per-entry runtime state migrated to `ConfigEntry.runtime_data` (story P3).** Pre-P3 the integration stored its per-entry coordinator + client in `hass.data[DOMAIN][entry.entry_id]` — the legacy HA pattern that pre-dates Home Assistant's 2024+ convention. The legacy shape had two real costs: (1) every read site needed two dict lookups (`hass.data[DOMAIN][entry.entry_id]["coordinator"]`) with `Any`-typed values that `mypy --strict` (story P6) couldn't see through, and (2) the dict was a global mutable on `hass.data` rather than scoped to the entry's lifecycle, which made multi-entry installs harder to reason about. P3 introduces `custom_components/iaqualink_robots/types.py` with `IaqualinkData` (a frozen-shape `@dataclass` holding the coordinator) and the typed alias `type IaqualinkConfigEntry = ConfigEntry[IaqualinkData]`. `__init__.py::async_setup_entry` now stamps `entry.runtime_data = IaqualinkData(coordinator=coordinator)` instead of writing to `hass.data`, and every platform (`button.py`, `sensor.py`, `vacuum.py`, `diagnostics.py`) reads `coordinator = entry.runtime_data.coordinator` — the client is accessible via the existing M15 `coordinator.client` property, so no second storage slot is needed. `async_unload_entry` was also reshaped to the HA-recommended order: unload platforms FIRST (so entities cleanly detach before any coordinator state goes away), then call `coordinator.cleanup()` — pre-P3 the order was inverted, which P10's lifecycle hardening already absorbed but P3's HA-native helper (`async_unload_platforms`) makes explicit. The 5 production read sites for `hass.data[DOMAIN]` are gone; a new `tests/test_static.py::test_no_hass_data_domain_access_in_production_code` regex-greps the package and fails CI if any production file (production-only — tests can still mention the legacy shape in xfail comments) reintroduces `hass.data[DOMAIN]` / `hass.data.get(DOMAIN, ...)` / `hass.data.setdefault(DOMAIN, ...)`. The two existing setup/unload tests in `tests/test_init.py` (both `@pytest.mark.xfail` due to a separate platform-fixture-wiring blocker that pre-dated this story) have their assertion bodies updated to the post-P3 contract — `entry.runtime_data.coordinator` instead of `hass.data[DOMAIN][entry.entry_id]["coordinator"]`. The diagnostics test helpers (`_entry()`, `_coordinator()`) were extended to wire the coordinator onto `entry.runtime_data` and the client onto `coordinator.client` so the existing 11 diagnostic tests continue to pass against the new read path. `__init__.py`'s previously-unused `asyncio` import is dropped (P3 removed the `asyncio.gather` over per-platform unload calls in favour of HA's `async_unload_platforms` helper). `button.py` and `sensor.py` no longer import `DOMAIN` from `const` (the only reach was for `hass.data[DOMAIN]`); `vacuum.py` keeps the import because it still uses `DOMAIN` in `unique_id` construction. **No user-facing behaviour change** — every entity continues to read the same coordinator data; the storage location is an implementation detail. The release-time risk is "third-party integrations or yaml-based code reading `hass.data["iaqualink_robots"]`" — extremely unlikely for a custom integration with no documented contract, and any such reader was already accessing private undocumented state.

- **`sensor.x_activity` / `sensor.x_fan_speed` / `sensor.x_status` raw state values are now lowercase machine keys, not title-cased English (story M11; breaking change for templates and automations).** Pre-M11 `AqualinkSensor.native_value` ran three inline display-maps over the cloud-side value before returning: `"cleaning"` → `"Cleaning"`, `"floor_only"` → `"Floor only"`, `"connected"` → `"Connected"`, and so on (15 raw-to-titled mappings across the three sensors). That title-casing bypassed the `translation_key` already declared on the entity — HA's frontend never saw the raw key, so the locale-aware `entity.sensor.<key>.state.<value>` translations P11 and P7 wired up were dead text on every install. It also forced user automations to write `is_state(sensor.x_activity, 'Cleaning')` with the awkward title-case (locale-fragile, non-obvious) and caused phantom history-graph "transitions" whenever the integration reloaded — the cached raw value would mismatch the next titled value and the recorder would log a transition that never happened. M11 strips the three display-maps from `native_value`; HA's frontend now looks up `entity.sensor.<key>.state.<value>` from `translations/*.json` (populated by P7) and renders the locale-appropriate text. **What you need to do on upgrade**: replace any template / automation references to the title-cased English values with the raw lowercase keys. The mapping is mechanical: lowercase + replace spaces with underscores. Examples — `is_state('sensor.x_activity', 'Cleaning')` becomes `is_state('sensor.x_activity', 'cleaning')`; `is_state('sensor.x_fan_speed', 'Floor only')` becomes `is_state('sensor.x_fan_speed', 'floor_only')`; `is_state('sensor.x_status', 'Connected')` becomes `is_state('sensor.x_status', 'connected')`. The full set of new raw keys (which now equal HA's history-graph state values and any template's `.state` access): activity = `cleaning` / `error` / `idle` / `returning` / `docking` / `paused`; fan_speed = `floor_only` / `wall_only` / `walls_only` / `floor_and_walls` / `smart_floor_and_walls`; status = `connected` / `disconnected` / `offline` / `online`. The Dutch-translation Dutch users see in HA's UI (`"Schoonmaken"` for `cleaning`, etc.) is unchanged — it's still rendered from the locale file, just now driven by `translation_key` instead of by Python-side title-casing. The `walls_only` (i2d waterline-only mode) vs `wall_only` (vr/cyclobat dedicated wall-scrub) distinction stays intact post-M11: both are distinct raw keys with distinct translations in `translations/*.json` (locked in by `tests/test_translations.py::test_fan_speed_states_include_both_wall_and_walls_variants`). New `tests/test_sensor.py` (20 tests) pins the new contract: every raw key value for each of the three sensors passes through `native_value` unchanged (15 parametrized cases — 6 activity + 5 fan_speed + 4 status), non-enum sensor keys (`battery_level`, `serial_number`, `model`) are unaffected, the resilient-cache behavior for transient `error_state` flags / `None` data / `"unknown"` values still preserves the last-known cached value, the `_last_value` cache updates on every successful read, and three regression guards (source-grep for `"cleaning": "Cleaning"`, `"floor_only": "Floor only"`, `"connected": "Connected"` shapes) plus one AST guard (forbid any title-casing helper method called from `native_value`) fail CI if a future "UX improvement" PR re-inlines the display maps under a different name. The user-facing rendered text in HA's frontend is unchanged for every locale that already had `entity.sensor.<key>.state.*` translations (all 9 locales post-P7).

### Added

- **Translations completeness + CI drift gate (story P7).** Pre-P7 `strings.json` (the canonical schema HA's developer tools and hassfest validate against) only contained the `config`, `options`, and `services` sections — the entire `entity.*` block was missing even though all 9 locale files (`cs`, `de`, `en`, `es`, `fr`, `it`, `nl`, `pt`, `sk`) carried it independently. A maintainer adding a new sensor / button / state could ship to one locale and silently skip others; CI had no gate against that drift. Worse, the three sensor-state substructures users actually see in the UI (`entity.sensor.activity.state.*` for cleaning/idle/error/returning/docking/paused, `entity.sensor.fan_speed.state.*` for floor_only/wall_only/walls_only/floor_and_walls/smart_floor_and_walls, `entity.sensor.status.state.*` for connected/disconnected/online/offline) lived only in `nl.json`. The other 8 locales rendered raw keys for those values — a Dutch user saw "Schoonmaken" while a Czech / German / French / Spanish / Italian / Portuguese / Slovak user saw the raw key text. P7 closes both gaps: `strings.json` is now the single source of truth for every key any locale uses (150 flattened keys, up from 73), the three sensor-state substructures land with English-fallback text in `strings.json` and in all 8 non-Dutch locales (Dutch `nl.json` already had native translations and keeps them — only the new `already_in_progress` abort key landed there as English fallback), and the C2-review absorbed `config.abort.already_in_progress` (the HA-core abort raised when a concurrent flow lands on the same `unique_id`) is now defined in `strings.json` and across all 9 locales so users see human-readable text instead of the raw key. The `wall_only` (vr/cyclobat dedicated wall-scrub mode) vs `walls_only` (i2d waterline-only mode) distinction is locked in by a dedicated test — both keys must coexist in `entity.sensor.fan_speed.state` because they map to two different cloud-side modes (see CLAUDE.md). New `tests/test_translations.py` (8 tests + 27 parametrize variants across the 9 locales) enforces: `strings.json` contains the required top-level sections (`config`, `entity`, `services`), the `entity` block has the three platforms (`sensor`, `button`, `vacuum`), the three sensor-state substructures exist with non-empty enumerations, `fan_speed.state` includes both `wall_only` and `walls_only` (regression guard against a future "fix the typo" PR), `config.abort.already_in_progress` is defined in `strings.json`, every locale file is valid JSON, **every locale file has exact key-shape equality with `strings.json`** (the drift gate — bidirectional check that catches both missing keys in a locale and dead keys in a locale that no longer exist in `strings.json`), every locale has non-empty `already_in_progress` text, and every locale has non-empty text for every sensor-state value. The drift gate is the centerpiece: any PR that adds a key to `strings.json` without backfilling all 9 locales (or removes a key from a locale without removing it from `strings.json` and the other 8 locales) fails CI with a clear `-/+` diff pointing at the exact offending key. The English-fallback convention for new keys (matching how `no_serial` was introduced for C2) is documented in the spec — native translations land via separate PRs without blocking the structural completeness. The deferred fan-speed value translations from P11's CHANGELOG entry are now landed via this story. **`exceptions.*` and `issues.*` top-level sections from the P7 spec are deferred** until the integration actually registers custom exception messages or Repair issues — the spec acknowledged these as conditional on other stories (P8 for issues; we have no custom user-visible exception messages today). The drift test will pick those up automatically once they're added to `strings.json` and any locale.

### Changed

- **Stepper add/reduce collapsed to one shared helper per layer (story R24, refactor + small behavior change).** Pre-R24 `AqualinkClient.add_fifteen_minutes` / `reduce_fifteen_minutes` were two near-identical ~95-line methods that differed only in the sign of the delta and the per-direction strings (success-message verb/preposition, error-message verb, log labels). Their coordinator-level wrappers (`AqualinkDataUpdateCoordinator.async_add_fifteen_minutes` / `async_reduce_fifteen_minutes`) were even more aggressively duplicated — ~80 lines each of identical debounce + browser_mod-or-persistent notification fallback + final refresh, parameterised only by which default text to use and which client method to call. Any bug fix in this area had to be applied twice and any drift between the two paths was a silent hazard. R24 replaces both pairs: `AqualinkClient._adjust_stepper(delta_minutes)` owns the cloud-side work — device-type guard, `_should_use_websocket()` check, 3 s rate limit, `fetch_status()` → compute → R23 `_build_state_request` set+clear pair → C4-cmd `_safe_post_command_refresh()` — and the two public methods become 1-line wrappers passing `+15` / `-15`. `AqualinkDataUpdateCoordinator._async_adjust_stepper(delta_minutes, client_method, success_message_default, failure_message_default)` owns the HA-side concerns — the 3 s debounce on `_last_timing_command`, the browser_mod-or-persistent notification fallback, and the final `async_request_refresh()` — and the two public coordinator methods become 4-line wrappers passing the sign, the client-method reference (`self.client.add_fifteen_minutes` / `reduce_fifteen_minutes`), and the per-direction notification text. **One small user-visible behavior change**: reducing the stepper when the cleaning cycle is already at 0 minutes now clamps to `0` instead of sending `{"stepper": -15}` to the cloud. Pre-R24 the cloud's behavior on a negative stepper was undefined — this was a latent corner case the UI never normally hit (the button still works for any `stepper > 0`; only a button mash at exactly `stepper == 0` would have triggered it). R24 makes the contract safe via `new_stepper = max(0, current_stepper + delta_minutes)`. AC#4's "~250 lines removed" target turned out to be optimistic for the same reason R23's AC#5 was — the shared helpers themselves take real LOC and the per-direction strings are now constructed at the helper's edges (`f"{'Add' if delta_minutes > 0 else 'Reduce'} {abs_delta} minutes"` etc.) rather than being hand-written constants. Actual net reduction is **~92 lines** (237 deletions, 145 insertions in `coordinator.py`). The qualitative win is the one R24's docstring identifies: bug fixes now apply once, the two directions can no longer silently drift, and the helper signature documents the symmetric-sign contract that pre-R24 was only enforced by copy-paste discipline. A new `tests/test_stepper.py` (14 tests) locks the contract: the 3 spec-mandated cases (`test_add_fifteen_increments_by_15`, `test_reduce_fifteen_decrements_by_15`, `test_reduce_at_zero_clamps`), an AST-based delegation guard for both layers (the two client wrappers must `await self._adjust_stepper(...)`; the two coordinator wrappers must `await self._async_adjust_stepper(...)`) that fails CI if a future PR re-inlines either direction, structural guards for the early-return branches (unsupported device type — vr/vortrax-only — websocket unavailable, rate-limited), the `_async_adjust_stepper` dispatch test (`async_add_fifteen_minutes` must call `client.add_fifteen_minutes` and not `reduce_fifteen_minutes` — defends against a sign-flip typo in a future refactor of the dispatch table), and a wrapper-LOC budget (≤ 8 top-level statements per wrapper) that fails CI if a future regression re-inlines the full ~95-line copy. Existing C4-cmd tests (`test_add_fifteen_minutes_awaits_coordinator_callback_exactly_once` / `test_reduce_fifteen_minutes_awaits_coordinator_callback_exactly_once`) continue to pass — the helper preserves the one-awaited-callback invariant C4-cmd locked in, and the AST guard in C4-cmd's `test_no_fire_and_forget_callback_create_task_in_command_methods` is unaffected because the new wrappers contain no `asyncio.create_task` at all. Manual smoke-testing on the user's VR robot post-merge would confirm the stepper UI still moves time correctly; the rest of the matrix (vortrax for the cloud-side path, all device types for the unsupported-device guard) is covered by the test fixture.

### Added

- **Coordinator lifecycle hardening (story P10 + review follow-up).** `coordinator.cleanup()` is the path Home Assistant takes on every entry reload — adding a config entry, removing one, or reloading after an OptionsFlow toggle. Pre-P10 it called `_stop_websocket_listener()` (which already cancelled + awaited the WS task) and `client._close_websocket()` (which already closed the persistent `_ws_session` + `_ws_connection`), but with two latent risks: (a) `_stop_websocket_listener` awaited cancellation **without a timeout** — a listener task that swallowed `CancelledError` would deadlock `async_unload_entry` indefinitely; (b) `vacuum._delayed_refresh()` was scheduled via `asyncio.create_task(...)` directly, making it a true fire-and-forget that would survive entry unload and try to refresh a torn-down coordinator. P10 closes both, and the **3-layer adversarial code review** (Blind Hunter + Edge Case Hunter + Acceptance Auditor) of the initial commit `c762b52` surfaced 11 follow-up patches — all applied in the same Unreleased window so the user-facing contract is the final, hardened shape: (1) `_stop_websocket_listener` wraps the await in `asyncio.wait_for(..., timeout=5.0)` and logs WARNING on timeout; (2) a new `_pending_tasks: set[asyncio.Task]` plus the public `schedule_task(coro)` helper (renamed from `_schedule_task` post-review — it's part of the cross-module coordinator-entity contract) tracks every fire-and-forget task, with self-deregistration via `add_done_callback`; (3) `_cancel_pending_tasks` cancels every task first then awaits the *batch* under a SINGLE 5 s `asyncio.wait` ceiling (review F2-HIGH: pre-review the per-task sequential 5 s wait would blow HA's 60 s integration-unload budget at N≥12 stuck tasks); (4) `cleanup()` is now idempotent — a `_cleanup_done` re-entry guard short-circuits the second call if HA races two unloads (review F4); (5) `client._close_websocket()` is also wrapped in `asyncio.wait_for(..., timeout=5.0)` — symmetric with the other two phases — so a stuck TLS shutdown can't strand unload (review F3); (6) done-task exceptions are explicitly retrieved via `task.exception()` so Python's `Task exception was never retrieved` GC warning doesn't fire from a `_delayed_refresh` that raised mid-flight (review F5); (7) `_cancel_pending_tasks` re-raises `CancelledError` when the *outer* `cleanup()` is being cancelled (e.g. HA hard shutdown), so the cancel propagates correctly instead of being silently swallowed (review F6); (8) `_pending_tasks` is `.clear()`-ed unconditionally at the end of cancellation so the post-state matches the log message (review F7). `vacuum.set_fan_speed` routes its `_delayed_refresh` through `coordinator.schedule_task(...)` — the only fire-and-forget producer in the integration today; an audit of all `asyncio.create_task` / `asyncio.ensure_future` call sites (Edge Case Hunter's report) confirmed no other consumers are missed. `tests/test_lifecycle.py` (11 tests post-review) locks every invariant: listener cancellation, owned-session closure, pending-task cancellation, listener-timeout tolerance with WARNING-log assertion, **close_websocket-timeout tolerance** with WARNING-log assertion (new), idle-safety, **re-entry-guard idempotency** (new), **done-task-exception retrieval** (new), **batched-cancel under single 5 s ceiling** (new — proves 8 stuck tasks finish in <2 s rather than 8×5 s = 40 s), `schedule_task` deregister with bounded `sleep(0)` polling (no longer a single-tick flake risk), and the 20× reload cycle that pins per-cycle `client._close_websocket.assert_awaited_once()` (replaces the previous, brittle `asyncio.all_tasks()` count comparison).
- **Diagnostics download (story P5).** Adds `custom_components/iaqualink_robots/diagnostics.py` so the integration's device card now exposes the **Download diagnostics** button in Home Assistant's UI. The exported JSON contains entry data, the cached coordinator state, and per-client counters (token-expiry, consecutive WebSocket failures, last update success) with credentials and PII redacted: `username`, `password`, `id_token`, `auth_token`, `refresh_token`, `api_key`, `first_name`, `last_name`, `email` are replaced with `"**REDACTED**"`, and the robot serial is partially redacted to the shape `R23***678` so cross-issue triage still works without exposing the full cloud-account pivot key. A new `tests/test_diagnostics.py::test_no_token_or_password_key_appears_unredacted` walks the rendered payload and fails CI if any future-added key matching `*token*` / `*password*` / `username` is ever emitted unredacted — the long-term net against drift when new credential fields are introduced.

### Changed

- **WebSocket state-request envelopes consolidated to one helper (story R23, refactor).** Pre-R23 the 11 command methods (`start_cleaning`, `stop_cleaning`, `pause_cleaning`, `return_to_base`, `clear_desired_state`, `set_fan_speed`, `_enter_remote_control_mode`, `_exit_remote_control_mode`, `_send_remote_command`, `add_fifteen_minutes`, `reduce_fifteen_minutes`) each contained one or more **near-identical websocket envelope literals** — 21 in total across the file, each 12–18 lines of nested-dict literal. The duplication hid two kinds of drift: **intentional** per-device-type (`cyclonext` uses the `robot.1` equipment key instead of `robot`; `cyclobat` nests start/stop/pause values under `main.ctrl` instead of writing them directly under `robot`; vr/vortrax stays flat) and **accidental** (vr/vortrax/cyclobat `return_to_base` and the stepper add/reduce envelopes shipped with their top-level fields in a different order than the surrounding sibling envelopes for the same device type — almost certainly a copy/paste accident from a different source envelope). The 21 sites now route through `AqualinkClient._build_state_request(action, desired, *, namespace=None, robot_key="robot")` which produces one canonical envelope: `{action, version, namespace, payload: {state: {...}, clientToken}, service, target}`. AC#4's "zero behavior change" was clarified with the user before implementation as **dict-content equality** (every field with the same value, key order ignored) rather than literal byte-for-byte equality — the iAqualink cloud is AWS-IoT-shaped and parses JSON into dicts, so transmit-time key order is unobservable. Every field present in the pre-R23 envelopes is still present with the same value; nothing was dropped, and per-device-type drift in `desired` content is preserved exactly. Net `coordinator.py` reduction: ~213 lines (353 deletions of envelope-literal lines + orphaned `clientToken` local assignments, minus the ~140 inserted lines covering the helper definition + 21 per-site one-line helper calls). The original AC#5 target of "~350 lines shorter" turned out to be slightly optimistic; the realised reduction is lower because the helper's docstring (which documents the AC#4 interpretation decision, the per-device-type drift it preserves, and the canonical envelope shape) is itself ~40 lines and the per-site calls average 4–5 lines each (the multi-arg helper calls for cyclobat / cyclonext occupy 5 lines when broken across keyword-arguments for readability). The qualitative win — adding a new command is now a 1-line `_build_state_request(...)` call instead of a 15-line copy-paste of an existing envelope — is fully delivered. Tests: a new `tests/test_envelope.py` covers 21 parameterized regression cases (one per pre-R23 envelope literal) asserting dict-content equality between the helper's output and the verbatim envelope literal that lived at that call site, plus 4 sanity tests pinning the canonical-shape contract (top-level field set, cyclonext `robot.1` key, namespace-defaults-to-device-type, and clientToken construction from `_id|_auth_token|_app_client_id`). The user has a VR robot for manual smoke-testing post-merge; cyclobat / cyclonext / vortrax coverage is carried entirely by the fixture-shaped tests against the dict-content invariants — the cloud accepted the pre-R23 envelopes for those device types in production, and the helper's output is dict-content-equal to those envelopes, so by transitivity the cloud will continue to accept them. Users of cyclobat / cyclonext / vortrax robots should report any regression via the issue tracker.
- **iAqualink cloud REST calls now share Home Assistant's connection pool (story L19, with post-review cookie-jar isolation).** Pre-L19 the four cloud REST sites — `send_login`, `get_devices`, `get_device_features`, `post_command_i2d` — each opened a fresh `aiohttp.ClientSession(headers=...)` per request, paid a TCP + TLS handshake, made one call, and tore the session down. With SCAN_INTERVAL at 3 s and the i2d poll path going through `post_command_i2d` on every cycle, that meant a fresh TLS handshake to `r-api.iaqualink.net` every 3 seconds for the entire lifetime of the integration — measurable cloud-side latency, measurable CPU on the HA host, and zero connection-pool reuse. The four sites now route through a new `AqualinkClient._cloud_session()` `@asynccontextmanager` helper. When the client has an `hass` reference, the helper constructs a per-call `aiohttp.ClientSession` that **reuses HA's pooled connector** (via `async_get_clientsession(hass).connector`, with `connector_owner=False` so HA still owns the connector's lifecycle) but has its **own private `aiohttp.DummyCookieJar`** — so cookies set on iAqualink responses (AWS-LB session affinity, Cognito) cannot persist on HA's shared cookie jar and bleed into other integrations consuming the same pool. The two leak axes the shared-session model exposes — auth headers, response cookies — are both defended: auth headers move from the session to the request (no session-level defaults), and the wrapper's private cookie jar silently drops every Set-Cookie. The retry-once paths in `get_device_features` and `post_command_i2d` rebuild the `Authorization` header per request and continue to pick up the rotated `_id_token` cleanly after a `force=True` re-auth, preserving the H9b semantics exactly. The helper has a defensive fallback for the two code paths that operate without a `hass` reference — the `AqualinkClient.discover_devices` classmethod used by the config-flow during initial setup, and the `AqualinkClient.send_login` calls in unit tests — where it constructs a one-shot ephemeral `aiohttp.ClientSession()` so existing tests' `aiohttp.ClientSession` monkeypatch pattern keeps working unchanged. The persistent websocket session (`_ensure_websocket_connection`) and the Vortrax web scraper (`_get_vortrax_model_from_web`) are explicitly NOT routed through the helper and remain isolated — WS because it owns long-lived connection state closed via `_close_websocket()`, and the scraper because it hits the third-party `zodiac-poolcare.com` domain and sharing HA's connector would leak HA's pool (and any iAqualink auth tokens routed through it by a future mis-implementation) to a third party. `__init__.py` now calls `client.set_hass(hass)` before the early-auth path so the integration's very first cloud call already takes the shared-pool branch rather than the no-hass fallback. **No user-facing behaviour change** — every cloud call still goes to the same endpoint with the same auth, just over a reused pooled connection. Tests in `tests/test_session.py` cover each of the four cloud sites (4 tests asserting `connector` reuse + `connector_owner=False` + `DummyCookieJar` + header-per-request), a dedicated cookie-isolation test that fires two cloud calls and asserts each wrapper has its OWN `DummyCookieJar` instance (no jar reuse across calls, no overlap with HA's shared jar), the ephemeral fallback path (test asserting an `aiohttp.ClientSession` without `connector` kwarg is constructed when hass is None), the scraper isolation invariant (test asserting `async_get_clientsession` is NEVER called from the scraper even when hass is set, and the constructed scraper session has no `connector` kwarg), and a static-grep guard that pins the count of direct `aiohttp.ClientSession(...)` constructions in `coordinator.py` to exactly 4 (helper's shared-wrapper + helper's ephemeral fallback + WS + scraper). Any future PR that adds a 5th cloud REST site bypassing the helper trips the guard in CI. The post-review cookie-jar isolation was added during a 3-layer adversarial code review of L19 — the Edge Case Hunter caught the symmetric cookie-leak risk that the original header-leak defence had missed.
- **Real-time websocket state is now pushed directly to entities — no HTTP refresh roundtrip per inbound shadow update (story C4-ws, with post-review patches for cyclonext coverage + auth propagation + baseline invalidation). 🔴 Critical refresh-storm fix.** Initial C4-ws commit (`9961a89`) shipped the push refactor. 3-layer adversarial code review (2026-05-18) caught three real issues addressed by three follow-up patches before any user-facing release: **F1 — auth-failure swallow in the listener's outer broad-except**: the per-message handler re-raised `ConfigEntryAuthFailed` correctly, but the outer `except Exception` wrapping the whole listener caught it (since `ConfigEntryAuthFailed` is an `Exception`) and demoted to WARNING — the listener task died silently and HA never saw the reauth signal from the WS path until the next polled cycle. Fixed by inserting `except ConfigEntryAuthFailed: raise` before the outer broad-except (same H10 precedent as the parallel C4-cmd review F1 patch). **F2 — cyclonext device family silently bypassed by the push path**: `_parse_ws_state_message`'s intake filter only read `equipment.robot`, but cyclonext stores its state under `equipment.robot.1`. Every cyclonext push hit the empty-robot check and was unconditionally dropped, forcing cyclonext owners onto polled-only refreshes at SCAN_INTERVAL=3s (versus the ~<200ms push latency every other family enjoyed). Compounded by `_state_signature` only reading vr/vortrax/cyclobat keys (`state`, `prCyc`, `main.ctrl`) — even if the filter had passed cyclonext through, the signature tuple was all-None for cyclonext, the first push fired, and every subsequent identical-all-None push was deduped. Fixed by extending the intake filter to accept `equipment.robot.1` as a fallback key and extending the signature tuple with cyclonext's actual state fields (`mode`, `cycle`, `errors.code`) — verified against `_update_cyclonext_robot_data` which reads exactly these keys. **F3 — stale baseline + signature after WS reconnect**: `_close_websocket` previously left `_last_ws_envelope` and `_last_applied_state_signature` populated across disconnect/reconnect cycles. New inbound deltas on the reconnected session would deep-merge onto a pre-disconnect baseline (silently propagating fields the delta didn't touch from stale state), and the first real post-reconnect push could be dedup-dropped if it happened to match the pre-disconnect signature. Fixed by nulling both in `_close_websocket` so the next `_ws_subscribe` re-seeds fresh state. Five new tests cover the three patches: cyclonext signature uniqueness across mode/cycle/error transitions, cyclonext intake-filter acceptance of `robot.1`, intake filter still rejects truly-empty robots, listener propagates `ConfigEntryAuthFailed` to HA, and `_close_websocket` clears both baseline and signature. Pre-C4-ws the websocket listener fired up to **two** unawaited `asyncio.create_task(self._coordinator_callback())` bursts per inbound `StateReported` message — one for stepper updates, one for state changes — each of which routed into the debounced `async_request_refresh` → `_async_update_data` → `fetch_status` HTTP roundtrip. A chatty cloud session therefore produced overlapping refresh fan-outs, AND a single message that touched both stepper and state simultaneously double-dispatched. A new `AqualinkClient.clear_desired_state()` POSTs `desired.equipment.robot.state = 0` via the existing `setCleanerState` action — the same payload `stop_cleaning` already used for vr, so the cloud-side contract is unchanged. The coordinator's `_async_update_data` detects a `cleaning → idle` transition between `_last_data` and the freshly fetched status — gated on `_pending_stop_reset` being unset so it only fires on **natural** completion, not on a user-initiated stop — and calls the clear. Currently scoped to `_device_type == "vr"`; other device types (vortrax / cyclobat / cyclonext) early-return until users report the same symptom. Regression test locks in that a `cleaning → idle` transition triggers the clear and a `cleaning → cleaning` does not.
- **`stop_cleaning` no longer double-sends the stop command on i2d_robot devices.** A stray return indent in the i2d_robot branch let control fall through after the first POST so the same `_post_command_i2d` invocation fired twice. The second call could no-op or, on a slow cloud round-trip, race with the device's transition to idle and surface as an unexpected error. The branch now returns cleanly after the single POST.
- **`reduce_fifteen_minutes` button no longer always reads `stepper=0`.** The button delegated to a nested dict lookup that hardcoded the stepper index to `0` instead of indexing on the current device's active stepper, so on devices where the stepper lives at a non-zero index the "−15 min" button would either no-op or apply to the wrong stepper. That field is reserved for **core** HA integrations (it signifies which HA version contains the integration); custom integrations declare HA-version compatibility via different mechanisms. The invalid field has been removed.

- **Config flow now supports reauthentication without losing entity history (story P4, bundled with H9b per the spec's pair-land Dependencies line).** When `ConfigEntryAuthFailed` fires (typically after a password change), Home Assistant shows a "Reauthentication required" notification. Clicking it presents a password-only form (email is shown as read-only context). Submitting a valid new password updates the entry in place — `unique_id`, entity_ids, area assignments, automations, and history are all preserved. Invalid credentials re-show the form with `invalid_auth`; network failures show `cannot_connect`. Translation keys added (English fallback for non-English locales; native translations welcome via PR).
- **Reauth flow is defensively robust against HA edge cases (story H9b sign-off).** The previous code used `assert self._reauth_entry is not None` to guard against a deleted-entry race, which is stripped under `python -O`. The flow now aborts cleanly with `reauth_unavailable` (new translation key) if the entry has been deleted between dispatch and form display. The password form enforces a min length of 1 locally instead of round-tripping empty submits to Cognito. A failed `async_reload` post-update is now logged as a WARNING instead of an unhandled exception — HA retries on the next coordinator cycle.
- **Wrong fan-speed list exposed for cyclonext robots (issue #76).** The CNX 30 iQ, RE 4400 iQ, RE 4600 iQ and other cyclonext-family devices were showing a non-functional "Walls only" cleaning mode added in 2.4.2. Cyclonext only supports `floor_only` and `floor_and_walls` on the cloud side, so picking "Walls only" left the entity state forked from the device. The fan-speed-list assignment in `vacuum.py` now branches per `device_type` and mirrors the per-type `cycle_speed_map` in `coordinator.py::_set_other_fan_speed`. Also tightens cyclobat to expose all 4 modes it actually supports (`wall_only`, `floor_only`, `smart_floor_and_walls`, `floor_and_walls`) instead of an i2d-shaped 3-entry list that hid `smart_floor_and_walls` and offered the non-existent `walls_only`. i2d_robot is unchanged — `walls_only` (plural) is a genuine i2d "waterline only" mode (0x04), distinct from `wall_only` (singular) used by vr/cyclobat.
- **Fan-speed changes from Home Assistant no longer propagate to cyclonext robots (issue #76).** Independent of the wrong-list bug above. The cyclonext branch of `AqualinkClient._set_other_fan_speed` keyed its `cycle_speed_map` by display-name strings ("Floor only", "Floor and walls") even though `vacuum.py::async_set_fan_speed` forwards the snake_case translation key. The lookup never matched, `request` stayed `None`, and the command fell through to a silent `{"success": False, ..., "error": "No valid request generated"}` return — the HA entity recorded the new fan_speed locally but no cloud call was made, leaving the device on its previous cycle. Reproduced by the RE 4400 iQ user comment on issue #76. Map keys now match the snake_case translation keys the rest of the integration uses.

### Changed

- **Cycle-duration lookup extracted into a single bounds-checked helper (story C5).** Three sites in `coordinator.py` (VR / vortrax / cyclonext branches of `_async_update_data`) previously did `list(robot_data['durations'].values())[result["cycle"]]` — a positional lookup that raised `IndexError` on unexpected cycle codes, masked by the surrounding broad `except Exception`. Cycle-recovery still worked (the except branch set `cycle_duration = 0`), but the failure was silent. The new `AqualinkClient._resolve_cycle_duration(durations, cycle)` helper centralizes the lookup, explicitly bounds-checks the cycle code, and logs a DEBUG breadcrumb on out-of-range / non-int input. Behavior on the success path is unchanged (positional lookup over insertion-ordered dict — the cloud's `durations` key names are not currently documented; team review explicitly downgraded "switch to named keys" out of scope pending an API sample).
- **Device card `Model` field now shows the device-type family (`vortrax`, `cyclobat`, `i2d_robot`, …) when the cloud hasn't returned a specific model name (story P9).** Pre-P9 (and pre-review), i2d_robot device cards always displayed `Model: Hidden` and pre-discovery cards always displayed `Model: Unknown` because the model-resolution chain treated those placeholder strings as truthy. The post-review fix treats `"Unknown"` and `"Hidden"` as falsy so the device-type fallback (`client.device_type`) actually fires. End result: i2d_robot users see `Model: i2d_robot` instead of `Model: Hidden` on the device card.
- **All entity platforms (vacuum, sensors, buttons) now group under a single device card in Home Assistant (story P9).** A shared `build_device_info(coordinator)` helper in the new `custom_components/iaqualink_robots/device.py` returns a typed `homeassistant.helpers.device_registry.DeviceInfo` keyed on the robot's serial. Pre-P9 each platform built its own dict in-place with divergent `name` / `manufacturer` / `model` / `sw_version` values (button reported `<device_type>_<serial>` as name and a hardcoded `sw_version="1.0"`; vacuum reported `manufacturer="iaqualink"` lowercase and the **integration's** version string in `sw_version`, which surfaced misleadingly in HA's device card as if it were the robot's firmware). Now every platform: keys on the robot serial (falling back to `client.robot_id` mid-setup before discovery completes), reports `name=coordinator.title` (the friendly entry title) consistently — closing the M15-deferred button-vs-sensor device-name asymmetry — uses `manufacturer="Zodiac"` (the parent company; matches the API hosts in `const.py`), reads `model` live from `coordinator.data["model"]` with `client.device_type` as a meaningful fallback (`"vortrax"`, `"cyclobat"`, …) before the model fetch lands, and reports `sw_version` as `None` when the cloud hasn't told us the device firmware (rather than fabricating a value). Also drops the last private-attribute reach-through in `button.py` (`getattr(client, '_model', 'Unknown')`) — `model` now flows through the coordinator like all other device state. The static-grep test extended in M15 now also guards against re-introducing `client._model` reads.
- **`AqualinkClient` now exposes `device_type` and `serial` as public `@property` descriptors, and `AqualinkDataUpdateCoordinator` exposes `title` as a public `@property` + setter pair (story M15; post-review the coordinator side was converted from a raw public attribute to a property for symmetry with the client and to keep storage underscored against future HA base-class extension).** Every external read of `client._device_type`, `client._serial`, or `coordinator._title` across `button.py`, `sensor.py`, `__init__.py`, and the coordinator's own update/notification paths has been swapped to the public name. A new static-grep test (`tests/test_static.py`) locks the contract — adding a new reach-through to those private attributes is now a CI failure. Behavior is unchanged: the properties return the underscored storage and the title attribute holds whatever `__init__.py` set from `entry.title`. Tests that constructed MagicMock clients/coordinators now set the public names too (matches what production reads). Picks up three absorbed M12-review items (button.py `client._serial` TODO swap, missing property-shape lock-in test, and test-side `coordinator._title` reach-through).
- **Config flow now dedupes already-configured accounts after discovery, on a per-serial basis (story C6).** Previously, re-adding an iAqualink account configured against an already-known robot landed the user on the select_device step (or post-create unique_id collision), with a confusing message. The flow now collects the serials already configured for the submitted username (case- and whitespace-normalized), runs `discover_devices`, and filters the response: if every discovered robot is already configured locally the flow aborts cleanly with `already_configured`; if at least one is new (the **multi-robot-per-account** case), the new robot is offered via the single-device shortcut (or select_device when more than one new robot remains). Re-adding the same account on a single-robot setup now resolves to a clean abort instead of a confusing form step. Multi-robot-per-account users can add their second (third, …) robot just by re-running the user flow with the same credentials. The username form schema also now enforces `vol.Length(min=1)` to reject empty submits locally.
- **Defensive hardening bundle (story M19).** Ten small patches land together to tighten gaps surfaced in the C2, M12, and M17 code reviews. Highlights: config flow now rejects non-string `serial_number` values (int / list / dict) from the cloud API before they reach `async_set_unique_id`, complementing the existing whitespace-serial guard from C2; `async_setup_entry` lazily strips a stray `api_key` field that may persist after a backup-restore from before the M17 migration; and `async_migrate_entry` now returns `False` on failure and leaves the entry at its previous version so HA retries the migration on next setup rather than silently advancing past a half-applied state.
- **`_authenticate()` is now serialised under an `asyncio.Lock` (story H9b).** Two coordinator update cycles overlapping near token expiry can no longer both fire parallel Cognito refresh requests racing on `self._id_token` / `self._token_expires_at`. The second caller acquires the lock after the first refresh completes, sees a valid token via the double-check guard, and returns without re-authenticating. Picks up the concurrent-refresh race absorbed from the H9a review.
- **JWT-exp fallback warning is now rate-limited (story H9b).** A persistently-malformed Cognito token shape would previously fire `_LOGGER.warning("JWT exp claim missing or unparseable; falling back to 1h hardcoded expiry")` on every hourly refresh, flooding the operator log. The warning now fires once per process; subsequent fallbacks log at DEBUG so the diagnostic signal is still available without spam. Picks up the third absorbed H9a review item.
- **HACS minimum Home Assistant version raised from `2025.1.0` to `2026.3.0` in `hacs.json` (salvage from PR #96 by @FejbyK).** No current functional dependency on a post-2025.1 API; the bump signals active maintenance against a current HA core and aligns the floor with the HA versions actually being tested against in CI. **User impact:** users on HA cores older than `2026.3.0` will see this version greyed out / unavailable in the HACS UI and need to either upgrade their HA install or stay on 2.5.1. Bundled with the same commit (`8084ee4`): GitHub Actions `checkout@v3 → @v6`, README status badges, deletion of the legacy `info.md` HACS description (superseded by README), and staged-but-not-yet-active `custom_components/iaqualink_robots/brand/` PNG assets for a future submission to `home-assistant/brands`. The directory + domain rename in PR #96 was deliberately **not** salvaged — it needs an `async_migrate_entry` migration shim to avoid orphaning every user's existing entities, and is tracked as story C7 for separate work.

### Migration / manual cleanup notes

- **Pre-C2 duplicate entries are NOT auto-merged.** If you previously added the same robot twice (resulting in two config entries / two coordinators racing on the same cloud account), this release will NOT consolidate them. Both entries continue to run. To clean up: go to **Settings → Devices & Services → iAqualink Robots**, identify the duplicate entry, and **Delete** it manually. The remaining entry will keep operating.
- **Sensor entities with email frozen as unique_id are NOT migrated.** If you encountered the M13 regression (sensors named after your iAquaLink email rather than the robot serial), those existing entities keep their email-based unique_id. New entries created after this release use the serial. To clean up: open the entity registry for the affected sensors and rename them manually, or delete and re-add the integration.
- **Button unique_ids are auto-migrated on first restart after upgrade.** A one-shot migration walks the entity registry and rewrites legacy `<title>_<command>` button unique_ids to `<serial>_<command>`. No user action required in the common case — automation referencing the button `entity_id` continues to work. If a button somehow ended up with both shapes (rare: downgrade-then-upgrade scenarios), the migration leaves the legacy entry in place and logs a warning; resolve via **Settings → Devices & Services → Entities** by deleting the duplicate.
