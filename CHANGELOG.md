# Changelog

All notable changes to the iAqualink Robots Home Assistant integration are documented here.

## Unreleased

### Added

- **Diagnostics download (story P5).** Adds `custom_components/iaqualink_robots/diagnostics.py` so the integration's device card now exposes the **Download diagnostics** button in Home Assistant's UI. The exported JSON contains entry data, the cached coordinator state, and per-client counters (token-expiry, consecutive WebSocket failures, last update success) with credentials and PII redacted: `username`, `password`, `id_token`, `auth_token`, `refresh_token`, `api_key`, `first_name`, `last_name`, `email` are replaced with `"**REDACTED**"`, and the robot serial is partially redacted to the shape `R23***678` so cross-issue triage still works without exposing the full cloud-account pivot key. A new `tests/test_diagnostics.py::test_no_token_or_password_key_appears_unredacted` walks the rendered payload and fails CI if any future-added key matching `*token*` / `*password*` / `username` is ever emitted unredacted — the long-term net against drift when new credential fields are introduced.

### Fixed

- **Vacuum entity attributes are now a defensive copy of coordinator data, not a reference (story M14).** Pre-M14 `_handle_coordinator_update` did `self._attributes = data` — a direct reference assignment that the inline comment marketed as a "performance" optimisation but was actually a latent aliasing bug: mutating `self._attributes` (for example via the H7 `restored` flag in `extra_state_attributes`, or any future entity-side mutation) would propagate back into the shared `coordinator.data` dict that every other platform (sensors, buttons) reads from. The reference is now a shallow `dict(data)` copy. Behaviour-equivalent for users; defensive against subtle state-leak bugs. A new `tests/test_vacuum.py::test_m14_attributes_decoupled_from_coordinator_data` mutates `self._attributes` after the update and asserts `coordinator.data` is unchanged — locks the contract in.
- **The 7 `remote_*` and `*_fifteen_minutes` services now actually work (story C1a; behaviour change).** `services.yaml` has advertised `remote_forward`, `remote_backward`, `remote_rotate_left`, `remote_rotate_right`, `remote_stop`, `add_fifteen_minutes`, and `reduce_fifteen_minutes` for releases — but the integration never called `async_register_entity_service`, so an automation calling `service: iaqualink_robots.remote_forward` raised `ServiceNotFound`. C1a wires each service to the matching `async_*` method on the vacuum entity via `entity_platform.async_register_entity_service` in `vacuum.py::async_setup_entry`. **User-facing behaviour change**: automations that previously failed with `ServiceNotFound` now actually move the robot (forward / backward / rotate / stop) or adjust the cleaning time (add / reduce 15 minutes). If you had workarounds calling these via `vacuum.send_command` or templating, those workarounds should still work but are now redundant. The companion Repairs-flow notification (story P8) that would surface this behaviour change to the user is still pending — for now the CHANGELOG is the louder signal. A new `tests/test_vacuum.py::test_c1a_async_setup_entry_registers_all_seven_services` and `test_c1a_each_service_method_exists_on_vacuum_entity` lock in the registration shape so a future refactor that drops or renames any method fails CI rather than silently breaking the service surface.

### Changed

- **Integration domain renamed from `iaqualinkRobots` (camelCase) to `iaqualink_robots` (snake_case) — story C7. ⚠️ Breaking change: requires manual re-add on upgrade.** Pre-C7 the integration's `manifest.json` declared the camelCase domain `iaqualinkRobots` while the directory was the snake_case `iaqualink_robots/`. That mismatch triggered hassfest's `[MANIFEST] Domain does not match dir name` error on every CI run and deviated from Home Assistant's snake_case convention for every other integration (`media_player`, `binary_sensor`, `config_flow`, never `mediaPlayer`). C7 flips the manifest domain to `iaqualink_robots` and silences hassfest. Closes issue #97 (FejbyK). **What you need to do on upgrade:** an automatic in-process migration was investigated and prototyped during this work, but blocked by HACS's "one integration directory per repository" constraint (verified against HACS source: `get_first_directory_in_directory(self.tree, "custom_components")`). HACS will not deliver a legacy-domain stub directory alongside the renamed one, so existing users' config entries will orphan on first boot after upgrade — the integration disappears from **Settings → Devices & Services**, automations bound to entity_ids break, history detaches. To recover: 1) **Go to Settings → Devices & Services**, find the orphaned (greyed-out) iAqualink Robots entry, and **Delete** it. 2) Click **+ Add Integration**, search for **iAqualink Robots**, and re-enter your iAqualink credentials. Entity_ids will be re-created with the same shape (one vacuum, sensors, buttons per robot) but **entity history for the previous-entry's entities will be detached** — automations and dashboards that reference those entity_ids will continue working as long as you delete + re-add for the same robot serial (the unique_ids are stable across this), but the historical chart/long-term-statistics data will not be carried over. We'll re-evaluate an automatic migration path in a future release if HACS adds multi-integration-per-repo support or via the HA Repairs flow once we have a re-add hook to anchor it to.
- **Dead `hass.localize(...)` translation block removed from `_format_time_human` (story C3).** The coordinator's helper that builds the `"X Hour(s) Y Minute(s) Z Second(s)"` string for the `time_remaining_human` sensor previously wrapped its return value in three `self._hass.localize("component.iaqualinkRobots.entity.vacuum.time_format.<unit>")` calls inside a `try/except (KeyError, AttributeError)`. `hass.localize` is a frontend JavaScript API and does not exist on the Python `HomeAssistant` object — every call raised `AttributeError`, was silently swallowed by the except, and fell through to the English fallback. The localised branch never executed once since this code was introduced. The whole try/except is gone; the helper is now a one-liner returning the English string directly. Behaviour unchanged at the user-visible level. A new AST-based test (`tests/test_coordinator.py::test_no_phantom_localize_calls_in_package`) fails CI if a `.localize(` call is reintroduced anywhere in the package. Properly localising the time-unit suffixes (`"Hour(s)" → "Heure(s)" / "Uur(en)"`) is a follow-up — the correct path is HA's `translation_key` + `translations/*.json` mechanism (stories M11 + P7), not a custom localize wrapper.
- **Entity `available` now follows a wall-clock outage threshold instead of a fixed failure count (story H7; behaviour-impacting for automations).** Pre-H7 entity `available` properties (vacuum, sensors, buttons) only checked `coordinator.data is not None`; the coordinator served stale `_last_data` through its broad-except path until `_consecutive_failures` exceeded a hard count of 30, at which point HA marked the entity unavailable. Combined with adaptive polling (1.5 s ↔ 10 s), that meant a transient ISP blip of **45 s – 5 min** could disable user automations bound to the `available` state. H7 replaces the count gate with a wall-clock gate (`LONG_OUTAGE_THRESHOLD_SECONDS` = 30 minutes, exposed in `const.py` for tuning): entities stay `available=True` with the last-known status through any outage shorter than that, then flip to `available=False` once the threshold is exceeded. Two new coordinator properties surface the state — `is_long_outage` (consulted by entity `available`) and `is_serving_stale_data` (surfaced verbatim as the new `restored` extra-state-attribute on every entity). Power users can branch automations on the latter: `{{ state_attr('vacuum.x', 'restored') }}` is `True` whenever the integration is mid-outage but still has cached data, automatically flipping back to `False` on the next successful poll. **Behavioural note for users with `available`-based automations:** if you previously relied on `available` flipping to False within a couple of minutes of cloud disconnect, that no longer happens — pair with `restored` if you need the "stale data" signal sooner than 30 minutes.
- **Real-time websocket updates now route through HA's documented `async_request_refresh()` API (story H10).** Pre-H10 the websocket-callback `_handle_realtime_update` delegated to a custom `_immediate_refresh` method that (a) bypassed the `DataUpdateCoordinator` update lock by calling `_async_update_data()` directly, (b) manually iterated HA-private `self._listeners` to push entity updates, and (c) then *also* called the public `async_update_listeners()` two lines below — so the manual loop was redundant with the public call. The HA-private `_listeners` registry is internal to the `DataUpdateCoordinator` base class and may change shape on any HA minor release; the manual reach-through was a latent break risk. `_immediate_refresh` is deleted entirely; `_handle_realtime_update` now calls `await self.async_request_refresh()` which goes through the proper lock, invokes the same listener-notification path, and honours HA's default ~0.3 s request-refresh debouncer (which collapses rapid websocket-event bursts into a single refresh — a desirable property the old loop did not have). A new AST-based test in `tests/test_coordinator.py` walks every `.py` file in the package and fails CI if `self._listeners` reach-through is ever reintroduced.

### Fixed

- **Stop + immediate-restart no longer rolls the UI back to "idle" (story H6).** Pre-H6, when the user pressed stop and then immediately restarted cleaning, the live `activity == "cleaning"` from the next cloud poll was silently overwritten back to `"idle"` by the queued reset from `stop_cleaning`. The integration's state would lie about what the robot was actually doing until the *next* poll cycle re-fetched. H6 makes the apply state-aware: the new `AqualinkClient._apply_pending_stop_reset` helper consults `data["activity"]` and discards the reset (without clobbering live state) if the cloud has already moved past idle. Also defends against the "stale zombie reset" path — a queued reset that never landed (long disconnect, network drop) and would otherwise apply minutes later — by pairing the values dict with a `_pending_stop_reset_at` timestamp and expiring it after `PENDING_STOP_RESET_MAX_AGE_SECONDS` (10 s, tunable via the new constant in `const.py`). PR #94's natural-completion detector continues to work unchanged — the gate it uses (`not client._pending_stop_reset`) reads the same dict-or-None truthiness it always has; the timestamp is a private contract between `stop_cleaning` and `_apply_pending_stop_reset`.

## 3.0.0-beta.1 — 2026-05-15

First pre-release of the 3.x line. Major version bump signals two breaking-ish
changes (timestamp wire format, vacuum entity friendly-name derivation —
details under **Fixed** and **Migration / manual cleanup notes** below), 16
completed stories of foundation work (config-flow hardening, reauth flow,
shared device card, datetime hygiene, HA-2024+ entity naming), and a community
bug fix that stops VR robots from auto-restarting after a natural cycle.

**Pre-release** — HACS users on the default channel will not see this version;
toggle "Show beta versions" in the integration's HACS panel to install it.

**Minimum Home Assistant raised to `2026.3.0`** via `hacs.json`. Users on older
HA cores will see this version greyed out in HACS and need to either upgrade
HA or stay on 2.5.1. See the entry under **Changed** for rationale.

### Added

- **Config flow now rejects duplicate robots.** Adding the same robot twice aborts with `already_configured` instead of silently spawning a second coordinator (story C2).
- **Config flow now rejects empty/whitespace serials.** If the cloud returns a device with no usable serial number, the flow aborts with `no_serial` instead of creating an entry. Guards against the previous M13 regression where the user's email got frozen as the entity unique_id (story C2).
- `no_serial` translation added to `en.json` and copied as English fallback to all other locale files (cs, de, es, fr, it, nl, pt, sk). Native translations welcomed via PR.
- **Button entity unique_ids now derive from the robot's serial number** rather than the (mutable) integration entry title. Renaming the integration entry no longer forks duplicate button entities in the registry (story M12).
- **Removed the dead `api_key` field from config entry data.** The integration has always read the API key from a constant in `const.py`; the stored copy in `entry.data` was never used. Existing entries are silently auto-migrated to drop the field on next setup (story M17).
- **Token refresh now uses the real JWT `exp` claim from the iAqualink Cognito IdToken** instead of a hardcoded 1-hour window. If AWS ever issues tokens with a non-1h lifetime (per-pool setting), refreshes fire at the correct moment rather than burning cycles or running stale. A 60-second safety margin is subtracted to absorb clock skew between AWS and the HA host. On the rare path where the token is missing or malformed, a `WARNING` is logged and the integration falls back to `now + 1h − 60s` (story H9a).
- **CI parity test** that asserts the button command list in `button.py` stays in sync with the migration tuple `_M12_BUTTON_COMMANDS` in `__init__.py`. Adding a new button command without updating the migration tuple is now a CI failure rather than a silent drift hazard that would surface as duplicate registry entries on next user upgrade (story M18).
- **`Include seconds in time-remaining display` option** in the config flow (and via Settings → Devices & Services → Configure after setup). Default is on (preserves the historical seconds-precision behavior). Turning it off makes the `time_remaining_human` sensor update only when the minute ticks over, eliminating ~20× of activity-log entries during long cleaning cycles. The global `SCAN_INTERVAL` is unchanged so remote-control button responsiveness is not affected. Power users who want exact-seconds precision can still derive it from `estimated_end_time` minus `now` in a template.

### Fixed

- **VR robots no longer auto-restart for a second cycle after natural cleaning completion (issue #89, PR #94 by @flimzy).** On Polaris VRX IQ+ and other vr-family devices, finishing a cleaning cycle naturally (without a manual stop) left the iAqualink cloud's `desired.state = 1` (cleaning). The cloud detected the mismatch against the robot's reported idle state and dispatched a command to restart the robot, kicking off another full ~2-hour cycle immediately. A new `AqualinkClient.clear_desired_state()` POSTs `desired.equipment.robot.state = 0` via the existing `setCleanerState` action — the same payload `stop_cleaning` already used for vr, so the cloud-side contract is unchanged. The coordinator's `_async_update_data` detects a `cleaning → idle` transition between `_last_data` and the freshly fetched status — gated on `_pending_stop_reset` being unset so it only fires on **natural** completion, not on a user-initiated stop — and calls the clear. Currently scoped to `_device_type == "vr"`; other device types (vortrax / cyclobat / cyclonext) early-return until users report the same symptom. Regression test locks in that a `cleaning → idle` transition triggers the clear and a `cleaning → cleaning` does not.
- **`stop_cleaning` no longer double-sends the stop command on i2d_robot devices.** A stray return indent in the i2d_robot branch let control fall through after the first POST so the same `_post_command_i2d` invocation fired twice. The second call could no-op or, on a slow cloud round-trip, race with the device's transition to idle and surface as an unexpected error. The branch now returns cleanly after the single POST.
- **`reduce_fifteen_minutes` button no longer always reads `stepper=0`.** The button delegated to a nested dict lookup that hardcoded the stepper index to `0` instead of indexing on the current device's active stepper, so on devices where the stepper lives at a non-zero index the "−15 min" button would either no-op or apply to the wrong stepper. Lookup now mirrors `add_fifteen_minutes`'s symmetric path.
- **Integration now loads under newer Home Assistant cores that reject `extra keys not allowed @ data['homeassistant']` in custom-integration manifests.** Recent hassfest versions in current HA cores rejected the top-level `homeassistant` field in `custom_components/iaqualink_robots/manifest.json` as a hard error, refusing to load the integration. That field is reserved for **core** HA integrations (it signifies which HA version contains the integration); custom integrations declare HA-version compatibility via different mechanisms. The invalid field has been removed.
- **Remote-control button names now render in your Home Assistant UI language (issue #79, story P11).** The seven remote-control buttons (Forward / Backward / Rotate Left / Rotate Right / Stop / Add 15 Minutes / Reduce 15 Minutes) previously displayed in English regardless of HA's language setting because [`button.py`](custom_components/iaqualink_robots/button.py) had its `_attr_translation_key` assignment commented out and a hardcoded English `name` property took precedence. The fix uncomments the translation_key wiring, removes the legacy name property, and adds the class-attribute `_attr_has_entity_name = True` — HA now composes friendly names as `<device-name> <localized-button-name>` for every locale (cs / de / es / fr / it / nl / pt / sk, plus English fallback) using the translations already present in [`translations/*.json`](custom_components/iaqualink_robots/translations/). No translation strings were added; they were dead before.
- **Vacuum entity now follows HA 2024+ entity-naming (issue #79, story P11).** The main vacuum entity used the legacy `name` property pattern, surfacing a friendly name that bypassed HA's translation system. The vacuum class now declares `_attr_has_entity_name = True` + `_attr_name = None` (class attributes), and the legacy `name` property is gone. With `_attr_name = None` the entity is treated as the primary entity for the device — its friendly name is just the device-registry name (the user-typed entry title), no English suffix appended. The previously-set `_attr_translation_key = "fan_speed"` (which pointed at translation keys hassfest had rejected and HA's frontend didn't consume) is also removed. Fan-speed *value* translations (the "Floor only" / "Wall only" dropdown options) remain English-hardcoded in [`sensor.py::fan_speed_display_map`](custom_components/iaqualink_robots/sensor.py); doing those properly requires the `state_attributes.fan_speed.state.<value>` translation shape and is left to a follow-up. The vacuum constructor's `device_name` parameter is dropped (was only used by the removed `name` property); call sites simplified accordingly. **Heads-up for upgrades:** the vacuum entity's friendly name now derives from the device-registry name only (the config-entry title). If you previously customized the device-registry name via the HA UI without re-running the config flow, your vacuum entity's friendly name will update to whatever the device-registry name currently says. Fresh installs and users who never renamed the device card see no change.
- **TIMESTAMP sensors render "Unknown" on no-data / parse failure / never-cleaned, not a misleading "right now" instant (H8 review follow-up).** Post-H8 the three TIMESTAMP-classed sensors (`cycle_start_time`, `estimated_end_time`, `last_online`) emitted `dt_util.utcnow()` from three failure paths: the no-payload websocket branch wrote the string `"unknown"` (incompatible with `device_class=TIMESTAMP`); the four per-device-type exception-fallback blocks wrote `utcnow()` (made the sensor display "cycle started a few seconds ago" every poll during parse failures); and `stop_cleaning` reset values wrote `utcnow()` (made the sensor display "stopped at &lt;local time&gt;" which looked like a fresh start). All five paths now write `None`, so HA renders "Unknown" — the honest signal. A bonus `if timestamp > 0:` guard around the four `dt_util.utc_from_timestamp(robot_data['cycleStartTime'])` calls also prevents the "1970-01-01" cycle-start render for never-cleaned robots from being interpreted as "55 years ago" by HA's TIMESTAMP frontend. Additionally `add_minutes_to_datetime` now `raise TypeError` (not `assert`) on a naive datetime so the fail-fast guard survives `python -O` / `PYTHONOPTIMIZE=1` deployments.
- **Timestamps the integration produces are now timezone-correct (story H8; breaking change for templates that parse the previous naive-local ISO strings).** Every datetime the integration emits to Home Assistant — `cycle_start_time`, `estimated_end_time`, `cycle_end_time`, `last_online`, and the JWT-expiry/model-fetch-backoff internals — is now an aware-UTC datetime instead of a naive-local ISO string. The `cycle_start_time`, `estimated_end_time`, and `last_online` sensor entities now declare `device_class=TIMESTAMP`, so HA's frontend renders them in the user's local timezone instead of as a raw string. Remaining-time arithmetic in `_calculate_times` is now smooth across DST transitions (the prior naive `datetime.datetime.now() - cycle_end_time` math could jump by an hour at spring-forward / fall-back). All sites use `homeassistant.util.dt.utcnow()` / `dt_util.utc_from_timestamp()`; the `add_minutes_to_datetime` helper now asserts its input is tz-aware and fails fast at the helper boundary on a naive datetime. Existing entity *history* stored with the wrong offset is left as-is — new data accumulates correctly going forward. **Template authors**: if you have an automation that parses `states.sensor.<robot>_cycle_start_time.state` as an ISO string, the value now includes a `+00:00` UTC offset (and renders in local time via `device_class=TIMESTAMP`). Template comparisons against `now()` should use `as_datetime()` rather than string slicing.
- **Authentication failures now raise `ConfigEntryAuthFailed` so HA dispatches reauth (story H9b).** Previously, when the iAqualink cloud returned HTTP 401 on a REST call (`get_devices`, `get_device_features`, `post_command_i2d`), the exception bubbled into the coordinator's generic update-failure ladder and the integration kept retrying with stale credentials until `_consecutive_failures` exhausted — no "reauth required" notification ever surfaced. A new internal `AuthFailedError` marker is now raised on `response.status == 401`; the coordinator converts it to `homeassistant.exceptions.ConfigEntryAuthFailed` so HA stops polling and dispatches its reauth handler. The fragile `"401" in str(e)` substring check on the websocket reconnect path is replaced by a typed `aiohttp.WSServerHandshakeError.status == 401` check. Also: command-time WS 401s (in `set_cleaner_state` / `get_device_status`) now surface to reauth instead of being swallowed by the broad-except retry ladder, and `discover_devices` failures during config flow now show `invalid_auth` for credential errors (was previously misclassified as `cannot_connect`).
- **Cognito-side credential rejection now surfaces reauth (story H9b sign-off P1).** The auth endpoint itself (`send_login`) was previously the one credential-rejection path the 401-wrapper missed. When the user rotated their iAqualink password and the coordinator retried `_authenticate(force=True)`, Cognito returned a `NotAuthorizedException`-shaped body that `_authenticate`'s dict accesses fell over with `KeyError` — the coordinator broad-except absorbed it as transient noise and reauth never fired. `send_login` now raises `AuthFailedError` on response status 401, 403, or non-JSON content type; `_authenticate` translates any remaining unexpected-body-shape `KeyError`/`TypeError` to `AuthFailedError` as well. End result: a password change in the iAqualink app now always surfaces HA's reauth prompt.
- **REST 401s now retry once after a forced token refresh before tripping reauth (story H9b).** Matches the spec's Risks/Mitigation clause: a transient cloud 401 (stale token, brief Cognito hiccup) no longer immediately surfaces a "reauth required" notification. The retry uses `_authenticate(force=True)` to bypass the local recheck and force a real Cognito round-trip — necessary because a cloud-side revocation looks locally-valid until the response says otherwise. Only the second consecutive 401 raises `AuthFailedError`.
- **Config flow now supports reauthentication without losing entity history (story P4, bundled with H9b per the spec's pair-land Dependencies line).** When `ConfigEntryAuthFailed` fires (typically after a password change), Home Assistant shows a "Reauthentication required" notification. Clicking it presents a password-only form (email is shown as read-only context). Submitting a valid new password updates the entry in place — `unique_id`, entity_ids, area assignments, automations, and history are all preserved. Invalid credentials re-show the form with `invalid_auth`; network failures show `cannot_connect`. Translation keys added (English fallback for non-English locales; native translations welcome via PR).
- **Reauth flow is defensively robust against HA edge cases (story H9b sign-off).** The previous code used `assert self._reauth_entry is not None` to guard against a deleted-entry race, which is stripped under `python -O`. The flow now aborts cleanly with `reauth_unavailable` (new translation key) if the entry has been deleted between dispatch and form display. The password form enforces a min length of 1 locally instead of round-tripping empty submits to Cognito. A failed `async_reload` post-update is now logged as a WARNING instead of an unhandled exception — HA retries on the next coordinator cycle.
- **Wrong fan-speed list exposed for cyclonext robots (issue #76).** The CNX 30 iQ, RE 4400 iQ, RE 4600 iQ and other cyclonext-family devices were showing a non-functional "Walls only" cleaning mode added in 2.4.2. Cyclonext only supports `floor_only` and `floor_and_walls` on the cloud side, so picking "Walls only" left the entity state forked from the device. The fan-speed-list assignment in `vacuum.py` now branches per `device_type` and mirrors the per-type `cycle_speed_map` in `coordinator.py::_set_other_fan_speed`. Also tightens cyclobat to expose all 4 modes it actually supports (`wall_only`, `floor_only`, `smart_floor_and_walls`, `floor_and_walls`) instead of an i2d-shaped 3-entry list that hid `smart_floor_and_walls` and offered the non-existent `walls_only`. i2d_robot is unchanged — `walls_only` (plural) is a genuine i2d "waterline only" mode (0x04), distinct from `wall_only` (singular) used by vr/cyclobat.
- **Fan-speed changes from Home Assistant no longer propagate to cyclonext robots (issue #76).** Independent of the wrong-list bug above. The cyclonext branch of `AqualinkClient._set_other_fan_speed` keyed its `cycle_speed_map` by display-name strings ("Floor only", "Floor and walls") even though `vacuum.py::async_set_fan_speed` forwards the snake_case translation key. The lookup never matched, `request` stayed `None`, and the command fell through to a silent `{"success": False, ..., "error": "No valid request generated"}` return — the HA entity recorded the new fan_speed locally but no cloud call was made, leaving the device on its previous cycle. Reproduced by the RE 4400 iQ user comment on issue #76. Map keys now match the snake_case translation keys the rest of the integration uses.

### Changed

- **Cycle-duration lookup extracted into a single bounds-checked helper (story C5).** Three sites in `coordinator.py` (VR / vortrax / cyclonext branches of `_async_update_data`) previously did `list(robot_data['durations'].values())[result["cycle"]]` — a positional lookup that raised `IndexError` on unexpected cycle codes, masked by the surrounding broad `except Exception`. Cycle-recovery still worked (the except branch set `cycle_duration = 0`), but the failure was silent. The new `AqualinkClient._resolve_cycle_duration(durations, cycle)` helper centralizes the lookup, explicitly bounds-checks the cycle code, and logs a DEBUG breadcrumb on out-of-range / non-int input. Behavior on the success path is unchanged (positional lookup over insertion-ordered dict — the cloud's `durations` key names are not currently documented; team review explicitly downgraded "switch to named keys" out of scope pending an API sample).
- **Device card `Model` field now shows the device-type family (`vortrax`, `cyclobat`, `i2d_robot`, …) when the cloud hasn't returned a specific model name (story P9).** Pre-P9 (and pre-review), i2d_robot device cards always displayed `Model: Hidden` and pre-discovery cards always displayed `Model: Unknown` because the model-resolution chain treated those placeholder strings as truthy. The post-review fix treats `"Unknown"` and `"Hidden"` as falsy so the device-type fallback (`client.device_type`) actually fires. End result: i2d_robot users see `Model: i2d_robot` instead of `Model: Hidden` on the device card.
- **All entity platforms (vacuum, sensors, buttons) now group under a single device card in Home Assistant (story P9).** A shared `build_device_info(coordinator)` helper in the new `custom_components/iaqualink_robots/device.py` returns a typed `homeassistant.helpers.device_registry.DeviceInfo` keyed on the robot's serial. Pre-P9 each platform built its own dict in-place with divergent `name` / `manufacturer` / `model` / `sw_version` values (button reported `<device_type>_<serial>` as name and a hardcoded `sw_version="1.0"`; vacuum reported `manufacturer="iaqualink"` lowercase and the **integration's** version string in `sw_version`, which surfaced misleadingly in HA's device card as if it were the robot's firmware). Now every platform: keys on the robot serial (falling back to `client.robot_id` mid-setup before discovery completes), reports `name=coordinator.title` (the friendly entry title) consistently — closing the M15-deferred button-vs-sensor device-name asymmetry — uses `manufacturer="Zodiac"` (the parent company; matches the API hosts in `const.py`), reads `model` live from `coordinator.data["model"]` with `client.device_type` as a meaningful fallback (`"vortrax"`, `"cyclobat"`, …) before the model fetch lands, and reports `sw_version` as `None` when the cloud hasn't told us the device firmware (rather than fabricating a value). Also drops the last private-attribute reach-through in `button.py` (`getattr(client, '_model', 'Unknown')`) — `model` now flows through the coordinator like all other device state. The static-grep test extended in M15 now also guards against re-introducing `client._model` reads.
- **`AqualinkClient` now exposes `device_type` and `serial` as public `@property` descriptors, and `AqualinkDataUpdateCoordinator` exposes `title` as a public `@property` + setter pair (story M15; post-review the coordinator side was converted from a raw public attribute to a property for symmetry with the client and to keep storage underscored against future HA base-class extension).** Every external read of `client._device_type`, `client._serial`, or `coordinator._title` across `button.py`, `sensor.py`, `__init__.py`, and the coordinator's own update/notification paths has been swapped to the public name. A new static-grep test (`tests/test_static.py`) locks the contract — adding a new reach-through to those private attributes is now a CI failure. Behavior is unchanged: the properties return the underscored storage and the title attribute holds whatever `__init__.py` set from `entry.title`. Tests that constructed MagicMock clients/coordinators now set the public names too (matches what production reads). Picks up three absorbed M12-review items (button.py `client._serial` TODO swap, missing property-shape lock-in test, and test-side `coordinator._title` reach-through).
- **Config flow now dedupes already-configured accounts after discovery, on a per-serial basis (story C6).** Previously, re-adding an iAqualink account configured against an already-known robot landed the user on the select_device step (or post-create unique_id collision), with a confusing message. The flow now collects the serials already configured for the submitted username (case- and whitespace-normalized), runs `discover_devices`, and filters the response: if every discovered robot is already configured locally the flow aborts cleanly with `already_configured`; if at least one is new (the **multi-robot-per-account** case), the new robot is offered via the single-device shortcut (or select_device when more than one new robot remains). Re-adding the same account on a single-robot setup now resolves to a clean abort instead of a confusing form step. Multi-robot-per-account users can add their second (third, …) robot just by re-running the user flow with the same credentials. The username form schema also now enforces `vol.Length(min=1)` to reject empty submits locally.
- **Defensive hardening bundle (story M19).** Ten small patches land together to tighten gaps surfaced in the C2, M12, and M17 code reviews. Highlights: config flow now rejects non-string `serial_number` values (int / list / dict) from the cloud API before they reach `async_set_unique_id`, complementing the existing whitespace-serial guard from C2; `async_setup_entry` lazily strips a stray `api_key` field that may persist after a backup-restore from before the M17 migration; and `async_migrate_entry` now returns `False` on failure and leaves the entry at its previous version so HA retries the migration on next setup rather than silently advancing past a half-applied state.
- **`_authenticate()` is now serialised under an `asyncio.Lock` (story H9b).** Two coordinator update cycles overlapping near token expiry can no longer both fire parallel Cognito refresh requests racing on `self._id_token` / `self._token_expires_at`. The second caller acquires the lock after the first refresh completes, sees a valid token via the double-check guard, and returns without re-authenticating. Picks up the concurrent-refresh race absorbed from the H9a review.
- **JWT-exp fallback warning is now rate-limited (story H9b).** A persistently-malformed Cognito token shape would previously fire `_LOGGER.warning("JWT exp claim missing or unparseable; falling back to 1h hardcoded expiry")` on every hourly refresh, flooding the operator log. The warning now fires once per process; subsequent fallbacks log at DEBUG so the diagnostic signal is still available without spam. Picks up the third absorbed H9a review item.
- **HACS minimum Home Assistant version raised from `2025.1.0` to `2026.3.0` in `hacs.json` (salvage from PR #96 by @FejbyK).** No current functional dependency on a post-2025.1 API; the bump signals active maintenance against a current HA core and aligns the floor with the HA versions actually being tested against in CI. **User impact:** users on HA cores older than `2026.3.0` will see this version greyed out / unavailable in the HACS UI and need to either upgrade their HA install or stay on 2.5.1. Bundled with the same commit (`8084ee4`): GitHub Actions `checkout@v3 → @v6`, README status badges, deletion of the legacy `info.md` HACS description (superseded by README), and staged-but-not-yet-active `custom_components/iaqualink_robots/brand/` PNG assets for a future submission to `home-assistant/brands`. The directory + domain rename in PR #96 was deliberately **not** salvaged — it needs an `async_migrate_entry` migration shim to avoid orphaning every user's existing entities, and is tracked as story C7 for separate work.

### Migration / manual cleanup notes

- **Pre-C2 duplicate entries are NOT auto-merged.** If you previously added the same robot twice (resulting in two config entries / two coordinators racing on the same cloud account), this release will NOT consolidate them. Both entries continue to run. To clean up: go to **Settings → Devices & Services → iAqualink Robots**, identify the duplicate entry, and **Delete** it manually. The remaining entry will keep operating.
- **Sensor entities with email frozen as unique_id are NOT migrated.** If you encountered the M13 regression (sensors named after your iAquaLink email rather than the robot serial), those existing entities keep their email-based unique_id. New entries created after this release use the serial. To clean up: open the entity registry for the affected sensors and rename them manually, or delete and re-add the integration.
- **Button unique_ids are auto-migrated on first restart after upgrade.** A one-shot migration walks the entity registry and rewrites legacy `<title>_<command>` button unique_ids to `<serial>_<command>`. No user action required in the common case — automation referencing the button `entity_id` continues to work. If a button somehow ended up with both shapes (rare: downgrade-then-upgrade scenarios), the migration leaves the legacy entry in place and logs a warning; resolve via **Settings → Devices & Services → Entities** by deleting the duplicate.
